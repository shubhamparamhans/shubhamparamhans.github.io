API Security Case Study

This article documents a real-world fintech security vulnerability involving wallet recharge APIs, missing server-side validation, and insecure transaction acknowledgement flows.

Fooling the Startup of the Year

Hacking into Ola Wallet — by Shubham Paramhans

A few weeks ago I was working on some weekend fun project… I exactly don’t remember what that was. I was monitoring my phone traffic from a proxy server. While doing that I saw Ola API calls going from my phone (since I was booking a cab).

For those who don’t know about Ola, Ola Cabs is the biggest cab provider in India and also the biggest competitor of Uber.

After seeing those flashing binary data going from my system I forgot my weekend project and started tweaking and reverse engineering Ola API’s which eventually resulted in breaking their money transaction system and bam I was able to recharge my Ola wallet with any amount.

For a long time I was thinking about publishing this work on the web but I have been way too busy all this time.

It’s innate in humans — We all love a story. And if you’re looking for a cool tech story, this one could be for you.

Chapter 0 — Something is Not Right

I was working on a small side project in which I was monitoring my phone traffic using MITM Proxy.

While booking a cab using Ola I saw their API calls going from my phone to Ola servers. I was amazed to see that all the requests were simple HTTP requests and not even encrypted.

I was also surprised to see that there was no OAuth token mechanism or any other security mechanism being used in the API calls.

This made it very easy to replicate the API calls using tools like Postman or even Google Chrome.

I started playing with the API calls and was able to fetch all cabs near my location.

At that point I was sure that something is not right here.

Chapter 1 — Digging the Tunnel

After fetching cabs data successfully I thought what else can I do with these APIs.

I started thinking about creating a simple app which would book a cab automatically for me whenever I want. Something like — I just open the app and it books a cab for me with all my preferences set.

I started building that app and was able to book cabs successfully using Ola APIs.

After that, a crazy thought came into my mind — what if I could book cabs for free?

That was the moment when I started tracking Ola wallet recharge API calls.

I started monitoring the recharge process carefully and tried to understand the complete flow of money transaction.

After some observation I found that the complete wallet recharge process consisted of three simple steps:

  • Generating an order id
  • Making payment gateway calls
  • Sending acknowledgement of transaction to Ola servers

All these API calls were simple HTTP requests and data was being sent in plain text.

This made it extremely easy to analyze and replicate the requests.

I started digging deeper into the APIs to understand how order ids were generated and how Ola was validating successful transactions.

Chapter 2 — Fire in the Hole

After understanding the complete flow of wallet recharge process, I decided to try it out practically.

I generated an order id using Ola API and then skipped the payment gateway step completely.

Instead of making the payment, I directly sent the transaction acknowledgement request to Ola servers.

To my surprise, the wallet recharge was successful.

I received an SMS saying that my Ola wallet has been recharged.

At first I thought it could be a one-time thing or maybe a glitch.

So I tried the same thing again.

And again.

And again.

Every single time, the wallet was getting recharged successfully.

I then realized that Ola servers were not validating whether the order id was already used or not.

The same order id could be reused multiple times to recharge the wallet.

This was a critical flaw in their transaction system.

At that point I stopped experimenting further because things were already going out of control.

The Aftermath

After realizing how serious the issue was, I decided to stop experimenting further.

I am an ethical hacker and I believe in responsible disclosure, so I mailed Ola about the bug and explained the complete issue to them.

I also asked them whether they had any bug bounty program.

Initially, the response I got was very disappointing.

I was asked to explain the issue again and again to different people from their support team.

After a few follow-ups, someone from their technical team finally acknowledged the issue and told me that they will fix it.

I waited for a long time but there was no fix deployed.

The bug stayed live for weeks.

This made me realize how seriously some startups take security — even when money is involved.

Conclusion

This whole incident taught me a very important lesson.

No matter how big or well-funded a startup is, if the core architecture is weak, it can easily be exploited.

Some of the major issues in Ola’s system were:

  • No authentication or encryption on critical APIs
  • No server-side validation of order ids
  • Blind trust on client-side transaction acknowledgements

Security should never be an afterthought — especially when financial transactions are involved.
